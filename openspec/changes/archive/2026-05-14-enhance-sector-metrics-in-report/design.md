## Context

邮件报告当前展示的"关联板块"仅包含板块名称，缺乏实时市场数据。用户需要打开行情软件或搜索才能判断板块涨跌，报告的价值受限。

**当前数据流**：
1. `data_provider/base.py` 的 `get_belong_boards()` 获取股票所属板块（仅名称）
2. `src/analyzer.py` 将板块信息放入 `intelligence['belong_boards']`
3. `src/notification.py` 仅渲染板块名称

**现有可复用能力**：
- `get_sector_rankings()` 方法已返回板块涨跌数据，包含 `name` + `change_pct`
- 各数据源（AkShare、efinance、Tushare）已支持板块行情查询

## Goals / Non-Goals

**Goals:**
- 在关联板块数据中增加实时行情（最新价、涨跌幅、成交额）
- 邮件报告中以表格形式展示板块名称+涨跌幅

**Non-Goals:**
- 不修改"板块涨跌榜"独立区块的展示（已有涨跌幅）
- 不添加板块历史走势、分钟级行情等高阶数据
- 不支持港股/美股板块（A股板块行情体系独立）

## Decisions

### D1: 板块行情获取方案

**选择**：复用 `get_sector_rankings()` 的底层数据获取能力，为每个关联板块查询实时行情

**理由**：
- 避免重复请求：板块涨跌榜本身已获取全市场板块行情，只需从中筛选关联板块
- 降低延迟：实时行情查询有网络开销，单独查询 N 个板块会产生 N 倍延迟
- 各数据源的 `get_realtime_quotes(['行业板块'])` 一次返回全部板块行情

**备选方案**：为每个板块单独调用行情 API（被否决 - 延迟过高）

### D2: 数据增强时机

**选择**：在 `src/analyzer.py` 中完成板块数据的行情增强

**理由**：
- 数据增强属于分析阶段的职责，一次处理所有股票
- `notification.py` 保持纯渲染职责，不涉及数据组装

**备选方案**：在 `notification.py` 渲染时查询（被否决 - 渲染阶段不应有网络 IO）

### D3: 渲染格式

**选择**：表格形式展示，列为"板块名称 | 涨跌幅 | 最新价"

**理由**：
- 表格比纯文本更紧凑，适配邮件宽度限制
- 涨跌幅用颜色标记（+涨 -跌）提升可读性

## Risks / Trade-offs

- **[Risk] 数据源失败**: 板块行情获取可能超时/失败 → Mitigation: 保持现有降级逻辑，行情获取失败时仅展示板块名称
- **[Risk] 板块名称匹配**: 各数据源板块名称可能不一致 → Mitigation: 使用模糊匹配或统一名称映射表
- **[Risk] API 限流**: 额外行情请求可能触发限流 → Mitigation: 利用已有板块涨跌榜缓存，避免额外请求
