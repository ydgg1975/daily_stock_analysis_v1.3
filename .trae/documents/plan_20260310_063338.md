我将重新优化 `GEMINI_API_KEY` 的加载与配置逻辑，确保其在本地开发和 GitHub Actions 环境下均能可靠运行。

### 1. 增强本地环境的 `.env` 加载 (src/config.py)
- **问题**：当前的 `setup_env` 依赖相对路径 `Path(__file__).parent.parent / '.env'`，如果在不同目录下运行脚本（如 `tests/` 或 `scripts/`），可能找不到 `.env` 文件。
- **优化**：
  - 修改 `setup_env`，增加多路径搜索策略：优先搜索 `ENV_FILE` 环境变量指定的文件，其次搜索当前工作目录 `os.getcwd()` 下的 `.env`，最后回退到项目根目录。
  - 使用 `Path.resolve()` 获取绝对路径，并在加载成功时打印调试日志（仅显示路径，不显示内容）。
  - **关键点**：保持 `override=True`，确保本地 `.env` 修改能即时生效，覆盖可能过期的系统变量。

### 2. 确保 GitHub Actions 兼容性 (src/config.py)
- **机制**：GitHub Actions 通过 `env` 上下文将 Secrets 注入为系统环境变量。
- **兼容策略**：
  - 在 `_load_from_env` 方法中，保留 `.strip()` 处理（防止 Secrets 复制粘贴时带入空格）。
  - **无需修改 YAML**：现有的 Workflow 配置 `GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}` 是标准的且正确的。
  - **验证逻辑**：在 `Config` 初始化后，增加一段针对 GitHub Actions 环境的自检日志：如果检测到 `GITHUB_ACTIONS=true` 但 `GEMINI_API_KEY` 为空，明确输出警告提示“Secret 未配置或未映射到环境变量”。

### 3. 优化 Analyzer 初始化诊断 (src/analyzer.py)
- **问题**：`GeminiAnalyzer` 初始化失败时，错误信息可能被上层捕获或吞没。
- **优化**：
  - 在 `GeminiAnalyzer.__init__` 中，除了检查 Key 的有效性（前缀、长度），还增加对 `google.generativeai` 库导入和配置的 Try-Catch 块。
  - 如果 Key 校验失败，输出明确的 `ERROR` 日志，指明是“未配置”、“格式错误”还是“被占位符填充”。
  - 如果配置成功，输出 `INFO` 日志：“Gemini AI 初始化成功，使用模型: xxx”。

### 4. 执行计划
1.  修改 `src/config.py`：增强 `setup_env` 的路径搜索能力，增加 GitHub Actions 环境下的 Key 状态检查日志。
2.  修改 `src/analyzer.py`：完善初始化过程中的错误处理和日志输出。
3.  本地验证：运行 `python main.py --no-notify` 确认配置加载正常。
